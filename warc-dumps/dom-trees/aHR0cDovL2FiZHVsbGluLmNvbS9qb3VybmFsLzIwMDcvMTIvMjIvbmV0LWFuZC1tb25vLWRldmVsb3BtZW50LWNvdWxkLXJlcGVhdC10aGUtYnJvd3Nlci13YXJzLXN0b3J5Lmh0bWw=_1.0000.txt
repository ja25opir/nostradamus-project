<div class="journal-entry-wrapper post-text authored-by-abdullin category-linux category-mono">
<div id="item2501857" class="journal-entry">

  <div class="journal-entry-float-day"><span class="name">Sunday</span></div><div class="journal-entry-float-date"><span class="month">Dec</span><span class="date">23</span><span class="year">2007</span></div>
  
  
  
  <div class="journal-entry-text">
  
      
      <h2 class="title">
    
        
    
        <a class="journal-entry-navigation-current" href="/journal/2007/12/22/net-and-mono-development-could-repeat-the-browser-wars-story.html">.NET and Mono development could repeat the "browser wars"&nbsp;story</a>  
     
         
    
      </h2>
      
       <div class="journal-entry-tag journal-entry-tag-post-title"><span class="posted-on">   <img title="Date" alt="Date" class="inline-icon date-icon" src="/universal/images/transparent.png">Sunday, December 23, 2007 at 2:43</span> <span class="posted-in">in <span class="tag-element"><a rel="tag" href="/journal/category/linux"> <img title="Category" alt="Category" class="inline-icon category-icon" src="/universal/images/transparent.png">Linux</a></span>, <span class="tag-element"><a rel="tag" href="/journal/category/mono"> <img title="Category" alt="Category" class="inline-icon category-icon" src="/universal/images/transparent.png">Mono</a></span></span>  </div> 

    
  
      
      <div class="body">
  
        
        
        
          <p><em>As I’ve said in <a href="/display/Search?searchQuery=why+do+the+people+switch+from+microsoft+windows+to">the previous post</a>, Windows is gradually loosing the market to Linux systems. Such thing would obviously lead to the increased demand in software development for the Linux world.</em></p>

<p><em>**NB: **I’ll be&nbsp;talking right now about the desktop applications with the feature-rich UI and capabilities. That’s the Smart Clients. Web applications&nbsp;are partially immune to the&nbsp;problem of Windows-to-Linux transition (plus the efficient delivery and the maintenance of the feature rich UI is not something they are extremely good at).</em></p>

<p>Here are things&nbsp;that could possibly matter in the future for the desktop applications:</p>

<ol>
<li><p><a href="http://www.novell.com/products/desktop/">SUSE Linux Enterprise Desktop</a> (SLED) is going to be one of the frontier points in the Windows-to-Linux migration, since it is <a href="http://www.microsoft.com/presspass/exec/steve/2006/11-02NovellInterop.mspx">supported by both Novell and Microsoft</a>. This Linux system is also the most friendly to the Windows refugees (I’ve experienced that) and is supported by the major PC vendors (<a href="/display/Search?searchQuery=why+do+the+people+switch+from+microsoft+windows+to">see the previous post</a>).</p></li>
<li><p>The only rich and efficient development technology that is supported by Microsoft and Novel is .NET. It&nbsp;works natively under the Windows and with some limitations it could run under Linux or Mac systems via the <a href="http://www.mono-project.com/Main_Page">Mono</a>&nbsp;(see the <a href="http://www.mono-project.com/Supported_Platforms">list of the supported platforms</a>). And Mono is the Novell’s project. </p></li>
</ol>

<p>By the way, if you are deep into this new Silverlight thing –&nbsp;do you know that Microsoft has “<a href="http://www.dotnet.sys-con.com/read/461417.htm">announced</a> a deal with Novell to have Mono provide Silverlight support for non-Microsoft platforms”? That statement delivers some message about Microsoft’s expectations on the future&nbsp;importance of Linux and Mono. And this company should have some qualified analytics.</p>

<p>Consequences:</p>

<ul>
<li>There will be demand for&nbsp;the developers/teams that&nbsp;could <strong>efficiently</strong> deliver solutions&nbsp;capable of running in two worlds (or in three, if you count Mac). That’s not a trivial task, since every operating systems has its own quirks (remember the consequences of the <a href="http://en.wikipedia.org/wiki/Browser_wars">browser wars</a>?).</li>
<li>The market supply of the experienced&nbsp;development resources&nbsp;in this area is quite limited (and the entire knowledge&nbsp;area itself is&nbsp;quite specific and young) so the&nbsp;prices will be quite interesting (means “high”).</li>
<li>The applications blocks and other recipes&nbsp;being delivered&nbsp;by the <a href="http://msdn.microsoft.com/practices/">Microsoft patterns &amp; practices</a> group will favor solutions that&nbsp;add more flexibility to existing .NET solutions and simplify&nbsp;.NET-to-Mono transition. Novell would do the similar thing.</li>
<li>Desktop applications will start&nbsp;using the “Mono compliant” logo and it will bear the same message as the “W3C valid” image.</li>
<li><p>One of the big differences between .NET and Mono is in the UI. Currently developers could easily get astonishing UI by using components that hack deeply into the Windows API (P/Invokes and such). Unfortunately this experience is not easily portable to Mono. So there’s likely to be an effort to create some high level unified UI abstraction layer capable of rendering great UI on any OS (Windows Presentation Foundation?!). I just hope there will be no analogue for the ActiveX technology.</p></li>
<li><p>The most productive tool for the .NET development is Microsoft Visual Studio. Yet it is not ready for the Mono development. And <a href="http://www.icsharpcode.net/OpenSource/SD/">#develop</a>&nbsp;along with <a href="http://www.monodevelop.com/Main_Page">MonoDevelop</a>&nbsp;still feel quite&nbsp;immature (and there is nothing like Resharper there). So the Novell and Microsoft&nbsp;could&nbsp;be&nbsp;delivering some solid Visual Studio&nbsp;Mono packages.&nbsp;They should let developers become more efficient in developing Mono-compatible applications (or porting the existing ones). I’m talking not just about the “Execute under the Mono VMWare image” button, but rather about code analyzers (some that bring <a href="http://www.mono-project.com/MoMA">MoMA</a> capabilities to the new level), guidance packages and reference implementations.</p></li>
</ul>

<p>Consequences:</p>

<ul>
<li>Virtualization technology is going to stay popular these years.</li>
<li>Development machines will&nbsp;obviously require quite a number of hardware upgrades to support virtualization. This will corellate with the new FSBs getting higher, processors getting more cores,&nbsp; DDR3 and hybrid memory becoming more popular.</li>
<li>The actual development is going to get even more complicated (yes, the “quirks” factor&nbsp;again). So in addition to the simple unit tests, there will be more of testing of interactions:</li>
<li>Functional testing.</li>
<li>It will be a common thing for the integration servers have nightly runs of functional tests&nbsp;on multiple virtual machines.</li>
<li>Diagnostics and configuration validation modules will be more common in complex solutions.</li>
<li><p>There’ll be new component packages and development solutions that promise to make Mono development as simple as drawing a couple of boxes and assembling these together (these always do show up on any frontier).</p></li>
<li><p>There’s also the psychological factor – people will be more willing to invest in the software solutions that work in the both worlds or enable existing solutions to do that.</p></li>
</ul>

<p>So&nbsp;the next years are going to be fun. Do you see here the opportunities of turning these consequences to your advantage?</p>

<p><em>If interested, stay tuned as I’ll be talking more about the LIM architecture of <a href="http://landorsystems.com/">Landor</a> and the changes we plan to make in order to simplify its transition to Mono.</em></p>
